================================================================================
1. TUDA — КРОСС-ПЛАТФОРМЕННЫЕ ЯДРА (Агент 1)
================================================================================

СТАТУС: Архитектура готова для NMC4. 8 файлов, 4 backend.

СУЩЕСТВУЮЩИЕ BACKEND:
  - AVX2 (Intel/AMD): Vec8, MicroKernel 6×16, FMA
  - NEON-A57 (Baikal-M): Vec4, MicroKernel 4×8
  - NEON-A75 (Baikal-S): Vec4, MicroKernel 8×12
  - E2K (Эльбрус): Vec4, MicroKernel 4×4

ПЛАН ДОБАВЛЕНИЯ NMC4:
  1. TudaConfig.h — enum Arch::NMC4 + define TUDA_NMC4     [30 мин]
  2. TudaVec.h — Vec4 для NMC4 (копия E2K паттерна)        [1 час]
  3. TudaMath.h — элементные функции (std::exp/log/sin)     [30 мин]
  4. MicroKernel_4x4_NMC4.h — скалярный C, 16 аккумуляторов [2 часа]
  5. TudaBLAS.h — dispatch_microkernel() + NMC4 case        [30 мин]

БЛОКЕР: NMC GCC не векторизует C/C++ код. Скалярный microkernel даст те же 1%.
Для 10-100x нужны:
  - nmpp intrinsics (nmppmMul_mm_32f) — ЕСТЬ в SDK
  - Ассемблер NMC4 (ISA документация) — НУЖНА от Дмитрия
  - Или: обходной путь — вызывать nmpp для целых матриц вместо microkernel

================================================================================
2. NM CARD HARDWARE BACKEND (Агент 2)
================================================================================

РАБОТАЕТ:
  ✅ DLL loading (nm_card_load.dll), 11 PL_* функций
  ✅ Board init: Reset → LoadInitCode → GetAccess → LoadProgramFile
  ✅ DDR bump allocator (0x00340000 - 0x1FF00000, ~500MB)
  ✅ PCIe transfer: PL_WriteMemBlock / PL_ReadMemBlock (синхронный)
  ✅ Polling dispatcher: 8 opcodes (matmul, rmsnorm, softmax, silu, rope, elem_*)
  ✅ Watchdog на стороне ядра (mem[31] инкрементируется)
  ✅ Graceful shutdown (OP_EXIT → status=1 → CloseAccess)

НЕ РАБОТАЕТ / ОТСУТСТВУЕТ:
  ❌ Watchdog НЕ ЧИТАЕТСЯ хостом — нельзя отличить hang от slow compute
  ❌ Нет retry/recovery — timeout = exception, ядро не перезапускается
  ❌ Нет weight persistence — DDR allocator сбрасывается каждый op
  ❌ Нет пайплайнинга PCIe — upload → compute → download строго последовательно
  ❌ Multi-core НЕ интегрирован в C++ runtime (только Python скрипты)
  ❌ Нет error logging — PL_* коды ошибок не разбираются (PL_TIMEOUT, PL_AGAIN)
  ❌ Нет PL_SyncArray — используется DDR polling вместо официального API

РЕКОМЕНДАЦИИ ДЛЯ ДМИТРИЯ:
  Приоритет 1: Watchdog monitoring + emergency restart     [2-3 дня]
  Приоритет 2: Retry logic + error logging                 [2-3 дня]
  Приоритет 3: Weight caching в DDR (не сбрасывать)        [1 неделя]
  Приоритет 4: Миграция на PL_SyncArray                    [3-4 недели]

3. NMPP VECTORIZED LIBRARY (Агент 3)
================================================================================

ПОЛНЫЙ СПИСОК NMPP ФУНКЦИЙ ДЛЯ ML:

  МАТРИЧНОЕ УМНОЖЕНИЕ (главное):
    nmppmMul_mm_32f   — IEEE 754 float32, со stride параметрами  ← КЛЮЧЕВАЯ
    nmppmMul_mt_32f   — float32 с транспонированием B
    nmppmMul_mm_32s32s — int32 × int32 → int32
    nmppmMul_mm_8s64s  — int8 × int64 → int64 (квантизация!)
    + ещё 10 целочисленных вариантов (2s, 4s, 8s, 16s, 32s, 64s)

  МАТРИЦА × ВЕКТОР:
    nmppmMul_mv_32s64s — матрица × вектор (для bias add)
    + 8s, 16s варианты

  COPY/MEMSET:
    nmppmCopy_32s, nmppmCopy_8s, etc. — быстрое копирование
    nmppmCopyua_*/nmppmCopyau_* — aligned ↔ unaligned

  ЧЕГО НЕТ в nmpp:
    ❌ Activation (relu, sigmoid, tanh, softmax, gelu)
    ❌ Normalization (layernorm, batchnorm, rmsnorm)
    ❌ Loss functions
    ❌ Backward/gradient ops
    ❌ Optimizer kernels (SGD, Adam)
    ❌ Reduction (sum, argmax, topk)

МЕХАНИЗМ ВЕКТОРИЗАЦИИ:
  - 4 независимых FPU cores (FPU0–FPU3) параллельно
  - `rep vlen` — цикл без branch overhead
  - `.matrix` modifier — аппаратное ускорение матричных операций
  - Stride-based access: [ar0++gr0] — автоинкремент
  - Библиотека написана на АССЕМБЛЕРЕ NMC4

ТИПЫ ДАННЫХ NMC4:
  nm1 (1-bit, 64/reg), nm2s (2-bit, 32/reg), nm4s (4-bit, 16/reg),
  nm8s (8-bit, 8/reg), nm16s (16-bit, 4/reg), nm32s (32-bit, 2/reg),
  nm64s (64-bit, 1/reg), float32 (IEEE 754)

TOOLCHAIN СБОРКИ:
  cc1plus -quiet src.cpp -mnmc4 -mgas -O2 -o src_gas.s
  nmc-as-gnu -o src_gas.o src_gas.s
  nmc-ld -o program.abs startup.o src.o libnmpps-nmc4.a -T nm6408brd.lds -e start

  | Измерено GFLOPS            | 4.37        | 20,000     | 1:4,576  |
  | Утилизация                 | 0.85%       | 67%        | 79× хуже|
  | Matmul 32×64×64            | 60 ms       | 0.075 ms   | 800×     |
  | 1B LLaMA tok/s             | 0.1         | 40         | 400×     |

  | PCIe BW (actual/theoretical)| 0.96/2500 MB/s | 8000/16000 | —    |

АНАЛИЗ BOTTLENECK:
  ■ ГЛАВНЫЙ: Скалярный RISC код (0.85% утилизации) — НЕ железо!
  ■ PCIe НЕ bottleneck: 0.037% утилизации (0.96 MB/s из 2500 MB/s)
  ■ DDR не bottleneck: 32 GB/s внутри карты, используется <1%
  ■ NM Quad x1: МЁРТВ для inference (80 ms/layer загрузка)

С ВЕКТОРИЗАЦИЕЙ (ПРОГНОЗ):
  Скалярный → nmpp matmul: 10-100× ускорение
  4.37 GFLOPS → 50-100 GFLOPS (10-20% утилизации)
  60 ms/matmul → 2-5 ms/matmul
  0.1 tok/s → ~1 tok/s (1B LLaMA)
  3.17 часа → 20-30 мин (109K transformer)

ТЕКУЩЕЕ СОСТОЯНИЕ: 70% готовности

  РАБОТАЕТ:
    ✅ Device registration (PrivateUse1 = nmcard)
    ✅ Tensor allocation (NMCardAllocator, 64-byte aligned)
    ✅ Device transfer: to_nmcard() / nmcard_to_cpu() через memcpy
    ✅ 41 операция в эмуляторе (20 unary, 13 binary, 4 reduce, 3 linalg)

  НЕ РАБОТАЕТ:
    ❌ Autograd НЕ подключён к NMCard ops → градиенты не текут
    ❌ Shape ops (view, reshape, transpose) → принудительно CPU
    ❌ CrossEntropyLoss не диспатчится
    ❌ clone(), contiguous() не реализованы для NMCard
    ❌ ensure_hardware_init() — хрупкие относительные пути
    ❌ Multi-device не поддерживается (только 1 ядро)
