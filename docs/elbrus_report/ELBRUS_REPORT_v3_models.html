1. Что было сделано в этой сессии

1.1. Найден ROOT CAUSE — RoPE convention mismatch

До фикса RoPE применялся как interleaved пары (2d, 2d+1) для всех архитектур. Это LLAMA_ROPE_TYPE_NORM в llama.cpp, и работает только для llama / mistral. Архитектуры qwen / qwen2 / qwen3 / qwen3moe / gemma / gemma3 / phi3 / phi-2 / stablelm / falcon / gptneox требуют NeoX-style half-split — пары (d, d + head_dim/2).

Источник: llama.cpp/src/llama-model.cpp:9164 функция llama_model_rope_type:

case LLM_ARCH_LLAMA:    return LLAMA_ROPE_TYPE_NORM;   // interleaved (2d, 2d+1)
case LLM_ARCH_QWEN3:    return LLAMA_ROPE_TYPE_NEOX;   // half-split (d, d+half)
case LLM_ARCH_GEMMA3:   return LLAMA_ROPE_TYPE_NEOX;
case LLM_ARCH_PHI3:     return LLAMA_ROPE_TYPE_NEOX;
…

1.2. Сравнение hidden state стало диагнозом

Сначала через PT_DUMP_HIDDEN=1 сравнивали qwen3-4B (broken) и mistral-7B (working). Магнитуды совпадают:

qwen3-4B  Q4_K_M (broken):   emb std=0.008  L0=0.339  L5=0.76  L17=1.03  L35=5.88  final=3.23  max=48.4
mistral-7B Q4_K_M (working): emb std=0.0016 L0=0.0045 L1=2.81  L5=2.82  L17=3.17  final=5.99  max=151.9

Magnitude OK, но направление wrong — потому что Q/K вращали не те координаты, к которым были обучены веса. Это сразу указало на permutation / convention bug, не на quantization, не на eps, не на bias.

2.9. phi3.5-mini — FIXED 2026-05-03 (commit d9dce9e)

Bisect через PT_NO_QUANT_GEMV=1 (FP fallback path) дал решающий результат:

  • DEFAULT (quant_gemv): bom<unk>×40 мусор, 6.3 tok/s.
  • PT_NO_QUANT_GEMV=1 (FP path): «Moscow, the capital city of Russia, is a sprawling metropolis with a rich history and vibrant culture. It's known for its iconic landmarks like the Kreml» — связный английский, 0.2 tok/s (FP fallback ~30× slower).

Корень: load_quantized_mmap() (default load path) не обрабатывал phi3 merged tensors (attn_qkv.weight и ffn_up.weight rows=2*intermediate). split_quant_rows жил только в load_quantized_to_cpu (fallback path). После mmap для phi3:

  • layer.q_attn_q.valid = false (нет attn_q.weight в GGUF)
  • layer.q_attn_k.valid = false, q_attn_v.valid = false
  • layer.q_ffn_gate.valid = false
  • layer.q_ffn_up.rows = 16384 (вместо 8192) — это merged [gate; up] в одной матрице

forward_decode_cpu else-branch:

if (layer.q_attn_q.valid && layer.q_attn_q.cpu_data) {
    cpu_quant::cpu_quant_gemv(...);  // ← НЕ вызывается → sp.q_buf garbage
}

sp.q_buf, sp.k_buf, sp.v_buf оставались uninitialized memory → RoPE на мусоре → softmax NaN-like → argmax всегда падал в token 0 (<unk>).

Fix: split_from_mmap helper в load_quantized_mmap. Когда attn_q.weight отсутствует но attn_qkv.weight есть — malloc + memcpy слайсов из mmap region в Q/K/V owning buffers. Аналогично для ffn_up.weight rows=2*intermediate → split на gate+up.

  • EN: «Moscow, the capital city of Russia, is not only the political heart of the country but also a vibrant cultural and economic center. With its rich history that dates back to 1...» (40 / 11.4s = 3.5 tok/s)

3. Скорость: 11.4 baseline и почему 8.0

Конфигурацияtok/sRussian quality
per-tensor scale + interleaved RoPE + spec K=1 11.4broken для qwen3
per-tensor scale + NEOX RoPE + spec K=1 ~10.5 (estimate)работает
per-block scale + NEOX RoPE + spec K=1 8.0работает

Юзер выбрал quality > speed. Per-block scale (точнее квантизация активаций) даёт правильные скоры внимания на массивных активациях qwen3, но добавляет ~7% overhead per замеру. NEOX RoPE сам по себе без overhead — те же O(head_dim/2) операций, просто другая индексация пар.

Объём: ~150 LoC в gguf_model.h + ~50 LoC dispatch в forward path.
