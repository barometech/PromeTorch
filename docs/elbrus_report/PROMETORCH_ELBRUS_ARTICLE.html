PromeTorch на Эльбрус 8C2: инференс современных LLM на отечественной VLIW-архитектуре

PromeTorch (открытый источник)
Эксперимент 2026-04 .. 2026-05, репозиторий github.com/barometech/PromeTorch
Аннотация. Описывается реализация инференса 10 современных квантизованных трансформерных моделей (qwen3 1.7B/4B/8B/14B, qwen2.5-7B, mistral-7B, gemma3-4B, phi3.5-mini, llama3-8B, deepseek-coder-7B) на 8-ядерном VLIW-процессоре Эльбрус-8С2 (1.5 ГГц, 4 NUMA-узла, 125 ГБ DDR4) без использования сторонних backend-библиотек. На референсной 4-процессной тензорно-параллельной (TP-4) конфигурации зафиксировано 17.1 ток/с для qwen3-1.7B, 10.9 ток/с для qwen3-4B (≈ исходный baseline 11.4 ток/с при сохранении правильного русского), 8.5 ток/с для mistral-7B Q4_K_M, 6.7 ток/с для gemma3-4B и 6.4 ток/с для phi3.5-mini. Single-process путь работает на всех 10 моделях, включая qwen3-14B (1.5 ток/с). Сопоставление с llama.cpp v3 32-thread (numactl --interleave=all) даёт прирост ×3.1 .. ×6.3. Выявлены и устранены три глубоких дефекта корректности (phi3 merged-tensor mmap load, gemma3 TP-4 post-norms placement, deepseek legacy rope.scale_linear ключ); код opensource. Результаты сравниваются с тремя предшествующими работами для архитектуры e2k: llama.cpp-e2k (порт legacy LLaMA-1), Smart Engines (CV-инференс) и ggml-NCNN-демо.

1. Введение

VLIW-архитектура Эльбрус (e2k) разрабатывается АО «МЦСТ» с конца 1990-х; модель 8C2 (изд. 2018) имеет 8 ядер по 1.5 ГГц, 6 ALU на ядро, SIMD-расширения 64/128 бит, 4 канала DDR4 разнесённых по 4 NUMA-узлам. Компилятор LCC (≥ 1.29) — программный планировщик инструкций; rcps intrinsics (например __builtin_e2k_qpfmuls/qpfadds) дают 4-lane fp32 SIMD без AVX2. Современные LLM (qwen3, gemma3, phi3 — выпуск 2024–2025 гг.) на этой платформе ранее не запускались публично.

PromeTorch — header-only C++17 inference-библиотека (≈ 95 000 строк), реализующая дисковый путь GGUF Q4_K/Q5_K/Q6_K/Q8_0, in-memory mmap, dequantize, GEMV, вращение позиций (RoPE NORM/NeoX), KV-cache, attention (включая GQA и SWA), все нормы (RMS, post-norm), 4-процессное тензор-параллельное распределение через POSIX shared memory AllReduce и speculative decode. Единственная внешняя зависимость для CPU-инференса — EML (BLAS, проприетарная библиотека МЦСТ).

2. Сопутствующие работы для Эльбрус

2.1. llama.cpp-e2k (alexmihalyk23, 2023–)

Порт llama.cpp на e2k. Поддерживает legacy архитектуры LLaMA-1, Alpaca, Koala, GPT4All, Vicuna, Pygmalion-7B (все выпуска ≤ 2023). Заявленные показатели для 7B Q4_0:

  • 4 потока: 50 мс/токен ≈ 20 ток/с
  • 8 потоков: 44 мс/токен ≈ 22.7 ток/с
  • FP16 8 потоков: 123 мс/токен ≈ 8.1 ток/с

Современные архитектуры (qwen3, gemma3, phi3) не поддерживаются — нет NEOX RoPE, нет GQA с qk-norm, нет sliding-window attention, нет phi3-merged QKV/gate-up split. Распределённого инференса нет. Репозиторий — port мейнстрим-llama.cpp с патчем `llama-cpp-e2k.patch`.

2.4. MCST EML

Проприетарная библиотека МЦСТ (BLAS 1/2/3, LAPACK, обработка сигналов/изображений). Используется PromeTorch для линейной алгебры (cblas_sgemm); на e2k 8C2 4×node-local EML демонстрирует 1840 GFLOPS (92 % пика arithmetic peak) против 324 GFLOPS у дефолтного 32-core EML без NUMA-биндинга.

3. Архитектура PromeTorch

3.1. Общая схема

Поток для одного forward-pass на токен (single-process, OMP_NUM_THREADS=32): embedding → 28..40 трансформерных слоёв (RMSNorm → QKV-projection → RoPE → attention → output_proj → +residual → RMSNorm → gate*up → SiLU/GeGLU → ffn_down → +residual) → output norm → lm_head. Q4_K_M квантизация для весов; активации хранятся в fp32.

3.2. Quantization-aware GEMV

Узкое место — N=1 GEMV против Q4_K весов (≈ 2.5 ГБ для qwen3-4B). Реализованы два пути:

  • q4k_gemv_scalar — fp16 → fp32 преконвертация per-row, prefetch заголовков next-block (4 super-blocks ahead, ~576 байт лидирующего расстояния), inner-loop SoA для LCC SWP-планирования.
  • q8_soa4 repack (Round 4) — в момент загрузки конвертирует Q4_K в Q8 SoA-4 layout (4-way row interleaving, per-block fp32 d_w/dmin + int32 sum_q + int8 weights). Inner GEMV использует __builtin_e2k_qpmaddubsh (16-way uint8 → int16 dot) вместо нибл-распаковки.

3.3. Tensor Parallel-4 (TP-4)

Per-rank: numactl --cpunodebind=N --membind=N на 1 из 4 узлов; q_dim_local = q_dim/4, kv_dim_local = kv_dim/4, inter_local выровнен на granularity 256 (Q4_K super-block). AllReduce — bounded-spin SHM `/dev/shm/prometorch_ddp_29500` с per-rank slot для частичных сумм. PT_TP_GATHER=1 переключает row-parallel attn_output / ffn_down на gather + replicated GEMV (нужно для gemma3 post-norms).

6. Ограничения и обсуждение

OOM в TP-4 для 7B+ моделей. Q8 SoA-4 reпак требует ≈ 2 ГБ на ранг для 7B-весов; 4 ранга × 2 ГБ + per-rank scratch вылезают за 16 ГБ виртуальной памяти на ранг (явный лимит на текущей конфигурации 8С2). Single-process путь для 7B/8B/14B работает. Решение — расширение SoA-репака на K-slice TP-4 (вместо репликации) или снижение granularity до 16-way SoA (×0.5 памяти).

Качество генерации mistral вне TP-4-orchestrator. В первом эксперименте multi-step demo использовался Qwen-style chat template на mistral-модели, что привело к пустому output. Обнаружено переключением на qwen3-4B (template совпадает) или нативное --chat с правильным wrapper для каждой архитектуры.

qwen3-0.6B. Не запускается связно (`!!!!!`); обнаружено что проблема в capacity model'и, не в архитектурном коде (~5 % параметров qwen3-1.7B хватает на CoT, у 0.6B — нет). Не code-fixable.

Зависимость от EML. Проприетарная библиотека МЦСТ; на системах без EML PromeTorch fallback'ится на собственный SIMD-кернел (qpfmuls/qpfadds), что даёт ~30 % деградацию на dense-GEMM-путях. Полный self-contained путь без EML — out of scope.

7. Зачем это важно

Эльбрус — единственная платформа российской разработки с открытой поставкой и совместимостью с x86_64 binary на уровне эмуляции/перекомпиляции. Запуск современных LLM (qwen3, gemma3, phi3 выпуска 2024–2025) на этой архитектуре до настоящего эксперимента не демонстрировался публично; единственный известный порт LLM — llama.cpp-e2k — поддерживает только legacy архитектуры выпуска 2023 г. и ниже.

Практический эффект: Эльбрус-8С2 при 1.5 ГГц в TP-4 конфигурации обеспечивает 8.5–17.1 тк/с для 1.7–7B квантизованных моделей с сохранением правильного русского языка. В сопоставлении с llama.cpp v3 32-thread baseline (тот же hardware, fair config) PromeTorch даёт ×3.1–×6.3 ускорение для всех моделей кроме phi3.5-mini (×3.1; ниже ввиду Q5_K attn_qkv веса, которые пока не пере-упакованы в Q8 SoA-4). Это первая публикация фиксации связного русского текста на качественно новом классе моделей (≥ 2024) на e2k-VLIW.

8. Воспроизводимость

Репозиторий: github.com/barometech/PromeTorch (open source, без проприетарных компонентов). Сборка через LCC ≥ 1.29. Запуск (qwen3-4B TP-4 при сохранении русского):

PT_Q8_SOA=1 PT_PER_BLOCK_SCALE=1 PT_LM_HEAD_FP=1 PT_NO_FFN_SOA=1 \
