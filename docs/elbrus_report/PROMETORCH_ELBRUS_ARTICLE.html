PromeTorch — открытый проект, май 2026
Репозиторий: github.com/barometech/PromeTorch   ·   Лицензия: открытый исходный код   ·   Тестовая платформа: Эльбрус-8С2, 8 ядер, 1.5 ГГц, 4×NUMA, 125 ГБ DDR4
Описывается реализация инференса 10 квантизованных языковых моделей (qwen3 1.7B / 4B / 8B / 14B, qwen2.5-7B, mistral-7B, gemma3-4B, phi3.5-mini, llama3-8B, deepseek-coder-7B) на 8-ядерном VLIW-процессоре Эльбрус-8С2 без сторонних backend-библиотек, кроме проприетарной EML МЦСТ. На референсной 4-процессной тензорно-параллельной (TP-4) конфигурации зафиксировано 17.1 токенов/с для qwen3-1.7B, 10.9 токенов/с для qwen3-4B (≈ исходный baseline 11.4 при сохранении правильного русского языка), 8.5 токенов/с для mistral-7B Q4_K_M, 6.7 токенов/с для gemma3-4B и 6.4 токенов/с для phi3.5-mini. Single-process путь работает на всех 10 моделях, включая qwen3-14B (1.5 токенов/с). Сопоставление с llama.cpp v3 в режиме 32 потока и numactl --interleave=all даёт прирост ×3.1 .. ×6.3. Выявлены и устранены три глубоких дефекта корректности (phi3 merged-tensor mmap load, gemma3 TP-4 post-norms placement, deepseek legacy rope.scale_linear ключ парсера GGUF). Результаты сопоставлены с тремя предшествующими работами для архитектуры e2k: llama.cpp-e2k (порт legacy LLaMA-1), Smart Engines (CV-инференс) и ggml-NCNN-демо. Демонстрируется multi-step tool-call пайплайн (write_file через <tool_call>) для генерации трёх HTML-страниц.

1. Введение

VLIW-архитектура Эльбрус (e2k) разрабатывается АО «МЦСТ» с конца 1990-х; модель 8C2 (изд. 2018) имеет 8 ядер по 1.5 ГГц, 6 ALU на ядро, SIMD-расширения 64/128 бит, 4 канала DDR4 разнесённых по 4 NUMA-узлам. Компилятор LCC (≥ 1.29) — программный планировщик инструкций; встроенные функции (__builtin_e2k_qpfmuls, qpfadds, qpmaddubsh) дают 4-lane fp32 SIMD без AVX2. Современные языковые модели (qwen3, gemma3, phi3 — выпуска 2024–2025) на этой платформе ранее не запускались публично.

PromeTorch — header-only C++17 inference-библиотека (≈ 95 000 строк), реализующая дисковый путь GGUF Q4_K/Q5_K/Q6_K/Q8_0, zero-copy mmap, dequantize, GEMV, вращение позиций (RoPE NORM/NeoX), KV-cache, attention (включая GQA и SWA), все нормы (RMSNorm, post_attn_norm, post_ffw_norm), 4-процессное тензорно-параллельное распределение через POSIX shared memory AllReduce и speculative decode. Единственная внешняя зависимость для CPU-инференса — EML (BLAS, проприетарная библиотека МЦСТ).

Ключевые достижения (по результатам этого эксперимента):
  • 10 LLM с выпуском 2024–2025 запущены на e2k (ранее не было публично).
  • qwen3-4B TP-4: 10.9 ток/с с правильным русским — близко к 11.4 baseline.
  • ×3.1 .. ×6.3 ускорение vs llama.cpp v3 32-thread на той же платформе.
  • Open source, без проприетарных backend (только EML).

2.4. MCST EML

Проприетарная библиотека МЦСТ (BLAS 1/2/3, LAPACK, обработка сигналов/изображений). Источник: altlinux.org/Эльбрус/eml. Используется PromeTorch для линейной алгебры (cblas_sgemm); на e2k 8С2 4×node-local EML демонстрирует 1840 GFLOPS (92 % теоретического пика) против 324 GFLOPS у дефолтного 32-core EML без NUMA-биндинга — прирост ×5.7 от NUMA-aware распределения работы.

3. Архитектура PromeTorch

Рис. 2. Поток выполнения forward-pass на токен в PromeTorch (TP-4). Розовым выделены квантизованные веса (Q4_K), зелёным — межранговая коммуникация через POSIX shared memory.

3.1. Quantization-aware GEMV

Узкое место — N=1 GEMV против Q4_K весов (около 2.5 ГБ для qwen3-4B). Реализованы два пути:

  • q4k_gemv_scalar — fp16 → fp32 преконвертация per-row, prefetch заголовков next-block (4 super-blocks ahead, ≈ 576 байт лидирующего расстояния), inner-loop SoA для LCC SWP-планирования.
  • q8_soa4 repack (Round 4) — в момент загрузки конвертирует Q4_K в Q8 SoA-4 layout (4-way row interleaving, per-block fp32 d_w / dmin + int32 sum_q + int8 weights). Inner GEMV использует __builtin_e2k_qpmaddubsh (16-way uint8 → int16 dot) вместо нибл-распаковки Q4.

3.2. Tensor Parallel-4 (TP-4)

На каждый ранг: numactl --cpunodebind=N --membind=N на 1 из 4 NUMA-узлов; q_dim_local = q_dim/4, kv_dim_local = kv_dim/4, inter_local выровнен на granularity 256 (Q4_K super-block). AllReduce — bounded-spin SHM /dev/shm/prometorch_ddp_29500 с per-rank slot для частичных сумм. Флаг PT_TP_GATHER=1 переключает row-parallel attn_output / ffn_down на gather + replicated GEMV (нужно для gemma3 post-norms).

6. Ограничения

OOM в TP-4 для 7B+ моделей. Q8 SoA-4 репак требует около 2 ГБ на ранг для 7B-весов; 4 ранга × 2 ГБ + per-rank scratch вылезают за 16 ГБ виртуальной памяти на ранг (явный лимит на текущей конфигурации 8С2). Single-process путь для 7B / 8B / 14B работает. Решение — расширение SoA-репака на K-slice TP-4 (вместо репликации) или снижение granularity до 16-way SoA (×0.5 памяти).

qwen3-0.6B. Не запускается связно (!!!!! output); причина — capacity модели (около 5 % параметров qwen3-1.7B хватает на CoT, у 0.6B — нет). Не code-fixable.

Зависимость от EML. Проприетарная библиотека МЦСТ; на системах без EML PromeTorch fallback'ится на собственный SIMD-кернел (qpfmuls/qpfadds), что даёт около 30 % деградации на dense-GEMM-путях. Полный self-contained путь без EML — out of scope.

7. Зачем это важно

Эльбрус — единственная платформа российской разработки с открытой поставкой и совместимостью с x86_64 binary на уровне эмуляции / перекомпиляции. Запуск современных LLM (qwen3, gemma3, phi3 выпуска 2024–2025) на этой архитектуре до настоящего эксперимента не демонстрировался публично; единственный известный порт LLM — llama.cpp-e2k — поддерживает только legacy архитектуры выпуска 2023 г. и ниже.

Практический эффект: Эльбрус-8С2 при 1.5 ГГц в TP-4 конфигурации обеспечивает 8.5 .. 17.1 токенов/с для 1.7 .. 7B квантизованных моделей с сохранением правильного русского языка. В сопоставлении с llama.cpp v3 32-thread baseline (тот же hardware, fair config) PromeTorch даёт ×3.1 .. ×6.3 ускорение для всех моделей кроме phi3.5-mini (×3.1; ниже ввиду Q5_K attn_qkv весов, которые пока не пере-упакованы в Q8 SoA-4). Это первая публикация фиксации связного русского текста на качественно новом классе моделей (≥ 2024) на e2k-VLIW.
