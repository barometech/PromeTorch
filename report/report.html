3. Q8 SoA4 weight layout — корневой kernel под qpmaddubsh

3.1 Структура блока 176 байт

Q4_K_M веса репакуются при загрузке в Q8 SoA4 — 4-row interleaved INT8 layout, оптимизированный под __builtin_e2k_qpmaddubsh:

Per super-row block (4 N-rows × 32 K-elements = 176 байт):
  bytes  0..15  : 4× fp32 d_w     (per-row block scale = d × sub_scale_q4k)
  bytes 16..31  : 4× fp32 dmin_m  (per-row min      = dmin × sub_min_q4k)
  bytes 32..47  : 4× int32 sum_q  (per-row sum of int8 weights в этом блоке)
  bytes 48..175 : 128 байт byte-interleaved weights:
                  8 K-групп × 16 байт каждая, layout per K-group of 4 K-elem × 4 rows:
                  [r0[0..3], r1[0..3], r2[0..3], r3[0..3]]

3.2 Внутренний K-loop и арифметика fold

На каждый super-row group из 4 N-rows и K=32 элементов (один блок 176 B):

v2di acc_i32 = {0, 0};
// Inner: 8 K-групп × 4 элемента = 32 элемента всего
_Pragma("loop count(8)") _Pragma("ivdep")
for (int kg = 0; kg < 8; kg++) {
    v2di p16 = __builtin_e2k_qpmaddubsh(W_v[kg], A_v[kg]);  // 8×i16 partial
    v2di p32 = __builtin_e2k_qpmaddh(p16, ONES16);          // 4×i32 reduce
    acc_i32  = __builtin_e2k_qpaddw(acc_i32, p32);
}

// Per-block FP fold (5 ops): dot_signed = acc - 128*sum_q
//   fp_acc += scale_a * (d_w * dot_signed - dmin_m * sum_a_block)
v2di shift_v    = __builtin_e2k_qpmullw(sum_q_v, SOA4_SHIFT128);
v2di dot_signed = __builtin_e2k_qpsubw(acc_i32, shift_v);
v2di acc_f      = __builtin_e2k_qpistofs(dot_signed);
v2di term_w     = __builtin_e2k_qpfmuls(scales_v, acc_f);
v2di term_d     = __builtin_e2k_qpfmuls(dmins_v, sa_v);
v2di delta      = __builtin_e2k_qpfmuls(scale_a_v,
                  __builtin_e2k_qpfsubs(term_w, term_d));
fp_acc          = __builtin_e2k_qpfadds(fp_acc, delta);

Логика арифметики: qpmaddubsh делает попарное multiply-add 16 знаковых × 16 беззнаковых байт с насыщением; lanes 0,1 — row 0; lanes 2,3 — row 1; ... ; lanes 6,7 — row 3. Активация передаётся в формате byte-broadcast (4 байта повторены 4 раза в 16-байтном векторе) что позволяет получить 4 dot products одновременно — по одному per row — без horizontal reduce внутри K-loop. Затем qpmaddh схлопывает adjacent pairs i16 → 4 i32, по одному per row.

Активация квантуется один раз в начале (q8_soa4_quant_activation): absmax → scale_a → quant в uint8 с offset+128 → broadcast 4-tuples × 4 для прямой подачи в qpmaddubsh.

4. Дисассемблирование q8_soa4_gemv: peak VLIW packing

Команда: objdump -d --start-address=0x76800 --stop-address=0x77400 build_elbrus/examples/gguf/test_gguf_inference. Извлечён фрагмент inner loop:

   76c20:
  ldqp,0,sm %dr15, _f16s 0x1f0, %qpg21          ← slot 0 (load weight)
  qpmaddubsh,1,sm %qpg22, %qpg19, %qpg19        ← slot 1 (compute)
  aaurwd,2 %dr25, %aaincr2                      ← slot 2 (AAU update)
  ldqp,5,sm %dr15, _f16s 0x20, %qpr61           ← slot 5 (load weight)

   76c48:
  qpmaddubsh,1,sm %qpg26, %qpg23, %qpg20        ← slot 1 (compute)
  aaurwd,2 %dr24, %aaincr1                      ← slot 2
  qpmaddubsh,4,sm %qpg27, %qpg24, %qpg22        ← slot 4 (compute)
  aaurws,5 %r26, %aad0                          ← slot 5

   76cb8:
  ipd 3
  bap
  ldqp,0,sm %dr22, _f16s 0x1f0, %qpg28          ← slot 0 (load act)
  qpmaddubsh,1,sm %qpr34, %qpr32, %qpr27        ← slot 1
  ldqp,2,sm %dr15, _f16s 0x2b0, %qpg29          ← slot 2
  ldqp,3,sm %dr22, _f16s 0x1e0, %qpg30          ← slot 3
  qpmaddubsh,4,sm %qpr33, %qpr31, %qpr28        ← slot 4
  ldqp,5,sm %dr22, _f16s 0x1d0, %qpg31          ← slot 5

Результат анализа. В одной wide-инструкции LCC помещает:

  • 2× qpmaddubsh (slots 1 и 4) — основной compute
  • 4× ldqp (slots 0, 2, 3, 5) — параллельные loads weights+activations
  • 1× aaurwd (AAU update для array prefetch register pointer)

Итого 6 ops per cycle на 6-wide VLIW = peak instruction density. LCC 1.29 выжимает пиковую плотность инструкций из intrinsics. Это объясняет, почему все микро-оптимизации внутри inner loop (см. §7.2 — dual-accumulator unroll) дают null effect: некуда дальше упаковывать слоты.

Notation: ,N после opcode = ALC slot N (исполнительный канал). sm = silent mode (исполняется независимо от exception masking). Базовое описание исполнительных каналов VLIW — раздел "Архитектура Эльбрус v5" в системе команд МЦСТ.

5. Применённые оптимизации (lossless verified)

Все изменения проверены: diff сгенерированного output text с baseline = bit-identical (за исключением timing line логгера).

5.1 Q8 SoA4 repack под qpmaddubsh

Файл: torch/io/q8_soa_repack.h.

4-row interleaved INT8 layout, репак из Q4_K при загрузке модели. Активация квантуется в uint8+128 с broadcast layout. Inner K-loop использует qpmaddubsh + qpmaddh + qpaddw цепочку.

Эффект: +5.6 tok/s (3.8 → 9.4) против чистого Q4_K-dequant kernel.

5.2 Persistent broadcast ThreadPool с futex

Файл: c10/util/ThreadPool.h + c10/util/Futex.h.

Замена mutex+condition_variable (10 ms fork/join overhead на каждый parallel_for) на:

  • Один shared task descriptor (master-written, workers-read)
  • Atomic generation counter + futex_wake_all
  • Per-worker ack slots (cacheline-padded 64 B, false-sharing-free)
  • Master сам участвует в drain (extra worker для small jobs)

5 инвариантов (I1-I5) для предотвращения deadlock'ов прошлых ревизий.

Эффект: +0.5 tok/s (9.4 → 9.9), spawn cost 10 ms → 0.12 ms.

5.3 8 потоков на rank, drop NUMA replicate

После persistent pool sweet-spot переместился с 7 на 8 потоков на rank (полный NUMA-узел). PT_NUMA_REPLICATE=0.

Эффект: +0.7 tok/s (9.9 → 10.6).

5.4 Fused gate+up Q8 SoA4 GEMV (dual)

Функция q8_soa4_gemv_dual — один parallel_for на оба GEMV (gate и up FFN-проекции) с shared activation reads.

Эффект: +0.2 tok/s (10.6 → 10.8).

5.5 AVX2/e2k SIMD attention math

Q@K и V@scores имели #if defined(__AVX2__)-ветку для x86 dev, fallback на scalar. На Эльбрусе шёл scalar. Добавлен #elif defined(__e2k__) через qpfmuls + qpfadds (4-lane fp32, head_dim=128 = 32 итераций).

Эффект: +0.6 tok/s (10.8 → 11.4) суммарно в составе пакета 5.5+5.6+5.7+5.8.

5.6 Fused SiLU + Q8 quant_activation

Функция q8_soa4_silu_quant_activation_fused: 5 проходов старого пайплайна (SiLU → absmax → quant → broadcast → sum_per_block) объединены в 2 прохода. Убран per-token std::vector<uint8_t>(K) аллок. Lossless (та же арифметика).

Эффект: в шуме (±0.05 tok/s); cleanup, struct improvement.

5.7 Triple-fused QKV GEMV

В TP-4 forward Q+K+V раньше делались как 3 раздельных q8_soa4_gemv = 3 parallel_for spawn'а × 36 layers. Заменено на один q8_soa4_gemv_triple.

Эффект: attn_phase 20.81 → 20.39 ms (−0.4 ms, в шуме на полной сумме).

5.8 RoPE cos/sin cache + scores buffer reuse

Две lateral-находки от агента-аудитора кода:

  • past_len идентичен для всех 36 слоёв одного decode-шага → rope_precompute вызывается 1 раз вместо 36, результат кешируется в TPParallelState.rope_*_cache.
  • std::vector<float>(total_seq) для attention scores аллоцировался внутри 8-head × 36-layer цикла (288 alloc/free на токен). Заменён на persistent tp_.scores_buf.

Эффект: в шуме; cleanup.

6. Опт-ин lossy режим PT_LAYER_SKIP

Static skip-list через переменную окружения PT_LAYER_SKIP="i,j,k,...". Каждый skipped layer = identity residual: x_buf[cur] остаётся без изменений (нет attention/FFN block contribution). KV cache не заполняется и не читается для этой позиции — consistent across tokens (skip фиксирован для всех decode steps).

Skip применяется только в decode-фазе (флаг tp_.in_decode_phase), не в prefill — для accurate prompt encoding. Default OFF.

7. Эмпирически опровергнутые гипотезы

Все попытки ниже не дали обещанного speedup на E2K v5 + LCC 1.29. Раздел особо важен для МЦСТ: фиксирует, что НЕ работает на этой архитектуре, с указанием корневой причины и микробенч-данных.

7.1 Q4 SoA4 (4-bit packed) — провал на SSE2 emulation

Файл: vliw_mission/e2k_vnni/q4_soa4_microbench.c.

Цель: уменьшить DDR bandwidth в 2× через 4-битное хранение weights. Inner kernel требует unpack of nibbles перед qpmaddubsh:

// SSE2 nibble unpack pattern:
__m128i packed = _mm_load_si128(qs);
__m128i lo     = _mm_and_si128(packed, MASK_0F);    // 8 i8 low nibbles
__m128i hi     = _mm_and_si128(_mm_srli_epi16(packed, 4), MASK_0F);
__m128i exp    = _mm_unpacklo_epi8(lo, hi);         // interleave

Результат: 10.82 ms = 9× медленнее Q8 SoA4 (1.21 ms). Корень: LCC транслирует _mm_srli_epi16 и _mm_unpacklo_epi8 через scalar shifts вместо native E2K instructions. SSE2 эмуляция не оптимальна для unpacking nibbles. Hand-asm на E2K мог бы разблокировать, но нужны native intrinsics (см. §8.2 рекомендации).

7.2 Dual-accumulator unroll в k-loop — null effect

Гипотеза: разорвать dependency chain на acc_i32 через 2 параллельные acc chains, дав 2× ILP на 6-wide VLIW.

// Было: 1 chain
v2di acc_i32 = {0, 0};
for (kg = 0; kg < 8; kg++) acc_i32 = qpaddw(acc_i32, qpmaddh(qpmaddubsh(...), ONES16));

// Стало: 2 chains, summed at end
v2di acc_a = {0,0}, acc_b = {0,0};
for (kg = 0; kg < 8; kg += 2) {
    acc_a = qpaddw(acc_a, qpmaddh(qpmaddubsh(W[kg],   A[kg]),   ONES16));
    acc_b = qpaddw(acc_b, qpmaddh(qpmaddubsh(W[kg+1], A[kg+1]), ONES16));
}
v2di acc_i32 = qpaddw(acc_a, acc_b);

Результат: 11.4 → 11.4 (null). Дисассемблер показал, что LCC уже автоматически unroll'ит исходный 8-iter loop с 8 separate destination registers (%qpg18, %qpg19, … %qpg27). Manual dual-acc дублирует то, что компилятор уже делает.

7.3 FMA fold через qpfnmas/qpfmas — недоступны на v5

Цель: сократить FP-fold с 5 ops на 3 ops через native FMA E2K:

// Было (5 ops):
v2di term_w = qpfmuls(scales, acc_f);
v2di term_d = qpfmuls(dmins, sa);
v2di delta  = qpfmuls(scale_a, qpfsubs(term_w, term_d));
fp_acc      = qpfadds(fp_acc, delta);

// Цель (3 ops через qpfnmas:= -(a*b)+c, qpfmas:= a*b+c):
v2di term_w = qpfmuls(scales, acc_f);
v2di diff   = qpfnmas(dmins, sa, term_w);    // = -(dmins*sa) + (scales*acc_f)
fp_acc      = qpfmas(scale_a, diff, fp_acc); // = scale_a*diff + fp_acc

Результат: compile error при -march=elbrus-v5:

error: built-in function "__builtin_e2k_qpfnmas" is not supported for current cpu mode

Документация /opt/mcst/lcc-home/1.29.16/e2k-8c2-linux/include/e2kintrin.h содержит declarations qpfnmas/qpfmas, но они помечены v6+. На v5 (8C2) недоступны. См. рекомендацию §8.2.1.

7.4 __builtin_prefetch на weight blocks — null

Добавлены prefetch hints на 2 блока вперёд (distance 352 байт) в inner loop:

for (int64_t b = 0; b < bpr; b++) {
    if (b + 2 < bpr) __builtin_prefetch(gp + (b + 2) * SOA4_GROUP_BYTES, 0, 1);
    ...
}

Результат: null (11.4 → 11.4). HW prefetcher на E2K v5 уже saturated на linear stride pattern весов в Q8 SoA4 layout.

7.5 Batched GEMM K=4 — РЕГРЕССИЯ 0.42×

Гипотеза: batch K=4 активаций amortize weight loads → density qpmaddubsh вырастет 2-4× → 2-4× speedup per-token. Это требовалось для интеграции speculative decoding в TP path.

Микробенч q8_soa4_gemm4_microbench.c на K=2560 N=2432:

Реализацияms/callКомментарий
baseline single-row q8_soa4_gemv (1 акт × 1 weight set)0.707—
batched K=4 q8_soa4_gemm (4 акт × 1 weight set)6.7324× работы
per-activation в batched1.6830.42× РЕГРЕССИЯ

Корневая причина. Дисассемблер показал, что 4 ldqp/cycle в исходном single-row уже распределены: ~2 weight + ~2 activation loads. Weight loads НЕ являются bottleneck. При K=4 activation ldqp scale линейно (16 → 64 loads/block). 4 acc chains saturate compute slots без дополнительного throughput. Memory pressure растёт квадратично с K.

Implication для МЦСТ. На e2k v5 для Q8 SoA4 GEMV нельзя amortize weight bandwidth через batching. Это compute-bound, не memory-bound, workload. Speculative decoding и EAGLE-style techniques из GPU literature не дадут wins на этой платформе без расширения SIMD ширины (см. §8.2.5).

7.6 Busy-spin ThreadPool (llama.cpp / Gemini DR pattern) — РЕГРЕССИЯ ×2

Gemini Deep Research Max + ссылки на llama.cpp паттерн:

"The llama.cpp developers explicitly abandoned lock/unlock mechanisms in favor of busy-waiting on atomic variables, as it directly bypasses the kernel and provides strictly superior performance for memory-bound and heavily synchronized LLM inference."

Реализация в c10/util/ThreadPool.h: workers spin 50000 итераций на atomic gen counter перед fallback в futex_wait. PT_TP_NOSLEEP=1 — pure spin без fallback.

Результат: 11.4 → 5.5 tok/s (×2 регрессия). Откачено.

Корневая причина. Наш TP-4 setup = 4 ranks × 8 workers = 32 spin'ующих threads, распределённых по 4 NUMA chips. Каждый gen_.load(memory_order_acquire) = read запрос в L3 cache. 32 threads × спин = постоянный поток L3 invalidation между chips через cross-chip interconnect. Это thrashes coherency и блокирует реальные compute операции на тех же чипах.

Эмпирический вывод. llama.cpp паттерн валиден для single-process с одним пулом workers (нет cross-NUMA coherency давления). На multi-rank TP setup с distinct NUMA scopes futex-based pool правильный даже несмотря на kernel overhead. Это противоречит "стандартному" совету HPC literature и важно задокументировать. См. рекомендацию §8.2.4 о cross-NUMA atomic latency docs.

7.7 PT_TP_GATHER (AllGather mode) — регрессия + не lossless

Existing flag, переключение output_proj и других секций с K-slice + AllReduce-sum на N-slice + AllGather-concat.

Результат: 9.0 tok/s регрессия + НЕ lossless. Output text расходится с baseline. Не использовать.

8. Заключение и рекомендации МЦСТ

8.1 Подтверждённый lossless потолок — 3 независимых угла

Lossless 11.4 tok/s на qwen3:4b Q4_K_M TP-4 на Эльбрус 8C2 с LCC 1.29 является подтверждённым peak. Каждый из трёх углов получен независимо и эмпирически:

  1. Дисассемблирование q8_soa4_gemv (§4) — peak VLIW packing 6 ops/cycle на 6-wide. Все слоты заняты. Compiler полностью утилизирует архитектуру.
  2. q8_soa4_gemm K=4 microbench (§7.5) — 0.42× регрессия per-act. Доказывает compute-bound (не memory-bound) природу workload'а на e2k v5.
  3. busy-spin pool probe (§7.6) — ×2 регрессия. Доказывает что NUMA coherency = реальный bottleneck для concurrency на 4-chip setup.

Speedup от 3.3 (Эльбрус llama.cpp baseline) до 11.4 = ×3.5. Достижение 13.8% от A100 PromeTorch на CPU-only Russian VLIW.

8.2 Что было бы полезно от МЦСТ для следующих generation / LCC release

8.2.1 Backport qpfmas/qpfnmas на v5

На e2k v5 (8C2) FMA-инструкции для quad-packed float32 объявлены в e2kintrin.h но недоступны для генерации. Доступны только на v6+. Для нашего kernel'а (§7.3) это блокирует сокращение FP-fold с 5 → 3 ops, что дало бы ~10% уменьшение fold-overhead = ~0.4 ms / token = +5% суммарно. Если железо v5 фактически поддерживает qpfmas (4-стадийный FP MAC через комбо ALU), компилятор должен разрешить эту инструкцию.

8.2.2 Native E2K instructions для 4-bit nibble unpack

Q4 квантизация (стандарт для LLM, 1/2 памяти от Q8) не работает через SSE2 эмуляцию (§7.1) — LCC транслирует _mm_srli_epi16 и _mm_unpacklo_epi8 через scalar shifts. На современном LLM inference Q4 — основной формат. Без native fast unpack:

  • Q4 SoA нельзя ускорить выше Q8 SoA на E2K (counterintuitive)
  • Bandwidth advantage Q4 потерян

Запрос на native intrinsics:

  • __builtin_e2k_qpsrlw_epi16(v, count) — параллельный right-shift 8 packed i16 с logical fill
  • __builtin_e2k_qpand_8b(a, b) — bitwise AND 16 байт (возможно уже есть как qpand?)
  • __builtin_e2k_qpunpcklbw_128(a, b) — interleave low 8 байт от двух 16-байтных операндов в 16 i16 lanes

8.2.3 LCC -fforce-loop-apb и APB documentation

В /opt/mcst/doc/lcc/ Array Prefetch Buffer (APB) практически не задокументирован для third-party разработчиков. Gemini Deep Research указывает APB как механизм обхода bottleneck'а на загрузках в inner loops. Если этот канал не используется LCC автоматически на наших intrinsics-loops, был бы полезен:

  • Pragma или флаг для принудительного включения APB
  • Документ с concrete examples
  • Реальные cycle counts для hit/miss scenarios

8.2.4 Cross-NUMA atomic latency documentation

На 4-chip 8C2 atomic operation между chips имеет latency ~10× от intra-chip (эмпирически — busy-spin pool падает до 5.5 tok/s). Документация:

  • Точные cycle counts для load-acquire / store-release across inter-chip и intra-chip
  • Рекомендации по placement атомиков (где выгодно spin, где futex)
  • Cache coherency protocol details для Эльбрус v5

помогла бы tuning concurrent algorithms заранее вместо empirical trial-and-error.

8.2.5 Wider INT8 dot-product instruction (32-byte)

qpmaddubsh на 8C2 работает на 16 байтах = 16 i8 mults + 8 i16 sums per cycle на slot. Аналог из x86 — VPDPBUSD на 64 байта (4× ширина). Для следующих gen / e2k v6 32- или 64-байтная версия qpmaddubsh удвоила бы / учетверила бы INT8 throughput для GEMV — это прямой путь к real speedup на LLM inference без алгоритмических изменений.

8.2.6 SVE-style scalable vector / variable-length

Опционально: scalable vectors (как ARM SVE) — позволили бы один исходный код адаптироваться под разную ширину SIMD от поколения к поколению без переписывания kernel'ов.
